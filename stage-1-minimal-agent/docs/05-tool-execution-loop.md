# 小目标 5：执行工具并把结果喂回模型（Python）

## 目标

实现 Agent 核心状态机，使模型在每次调用一个工具后观察真实结果，再决定下一步。

## 状态流程

```text
初始化消息
   ↓
请求模型决策 ←────────────────────┐
   ↓                              │
解析 JSON                         │
   ├── final ──> 返回答案          │
   │                              │
   └── tool_call                  │
          ↓                       │
      注册表执行工具               │
          ↓                       │
      格式化成功/失败结果 ──────────┘
```

## 实施计划

1. 计算 Agent 总截止时间。
2. 建立系统消息和用户消息列表。
3. 用 `for step in range(...)` 限制循环次数。
4. 每步调用一次 `llm.complete()` 并保存原始助手输出。
5. 解析决策；`FinalDecision` 立即返回。
6. 其余决策交给 `ToolRegistry.execute()`。
7. 用 `json.dumps(..., ensure_ascii=False)` 序列化工具结果。
8. 给工具数据加明确的开始、结束标记和信任边界提示。
9. 把结果作为用户角色消息追加，再进入下一步。
10. 返回 `AgentRunResult`，消息历史使用 tuple 保存。

## 错误策略

工具参数错误、文件不存在、未知工具和工具超时通常不是整个 Agent 的致命错误。注册表把它们转成 `ok: false` 结果交给模型，模型可以修正参数或向用户说明限制。

配置错误、LLM 网络错误和 Agent 总限制属于控制层错误，会终止任务并由 CLI 统一显示。

## 数据信任边界

读取的文件可能包含类似“忽略之前指令”的文本。系统提示和工具结果消息都声明工具内容只是数据，不能作为上层指令执行。这是 Stage 1 的基础防护，完整 prompt injection 评测属于 Stage 7。

## 验收标准

- [x] 模型可以零次或多次调用工具。
- [x] 每个工具结果都进入下一轮消息上下文。
- [x] 工具失败不会被标记为成功。
- [x] 最终结果包含答案、步骤数和消息快照。
- [x] 事件回调可观察步骤、协议错误、工具状态和完成。
- [ ] 使用真实模型检查完整执行 trace。
