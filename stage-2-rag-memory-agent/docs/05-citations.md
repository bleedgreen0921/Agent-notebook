# 小目标 5：在回答中给出来源或证据

## 目标

把“模型声称有引用”变成程序可以验证的契约，并由程序负责输出真实链接。

## 证据流

```text
工具返回 EvidenceSource
        ↓
EvidenceLedger 去重并分配 S1、S2
        ↓
工具结果把编号和片段交给模型
        ↓
模型 answer 使用 [S1]，citations 声明 S1
        ↓
CitationValidator 检查
        ↓
CLI 根据账本输出标题、locator 和 URI
```

## 实施计划

1. 所有事实型工具返回标题、URI、证据片段和可选 locator。
2. 以 `(URI, locator)` 去重来源。
3. 同一来源后续得到更长正文时更新证据，但保持编号稳定。
4. 最终输出协议要求 `answer` 和 `citations`。
5. 正文中使用 `[S1]` 等内联标记，并紧邻相关结论。
6. `citations` 与正文标记必须集合完全一致。
7. 所有编号必须能在本轮账本找到。
8. 已获得证据却没有任何引用时拒绝回答。
9. 模型不直接输出来源 URL，CLI 从账本输出，避免编造链接。

## 能验证与不能验证的内容

程序能够验证：

- 引用编号是否存在。
- 正文标记和 citations 数组是否一致。
- 来源 URI 是否确实来自工具结果。
- 输出链接是否与账本一致。

程序尚不能仅靠编号验证：

- 某句话是否真正被来源片段支持。
- 来源本身是否权威、最新或无偏见。
- 综合推断是否合理。

这些需要 Stage 7 的引用蕴含评测、来源质量规则和固定测试集。

## 对应代码

- `models.py`：EvidenceSource、RegisteredSource。
- `evidence/ledger.py`：登记、去重、编号。
- `evidence/validator.py`：确定性引用校验。
- `agent/protocol.py`：最终输出结构。
- `agent/loop.py`：校验失败后的修正循环。
- `main.py`：最终来源链接输出。

## 验收标准

- [x] 来源编号只能由程序分配。
- [x] 最终链接只能来自证据账本。
- [x] 不存在或不一致的引用会被拒绝。
- [x] 本地文件、数据库、搜索和网页结果都可形成证据。
- [x] 长期记忆不能形成事实引用。
- [ ] 在可运行环境中增加“答案是否受证据支持”的自动评测。
